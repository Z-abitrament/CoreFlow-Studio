# Pulse Counter Module

## Summary
The pulse counter module parses DSView/libsigrok4DSL CSV exports and converts
logic-level pulse edges into traceable quantity, rate, frequency-segment, and
calibration-preview data.

The first implementation is offline only. It does not open DSLogic hardware,
install drivers, or perform live capture. Future live capture should feed the
same edge/time model used by the CSV parser.

The Qt UI exposes this as `Modules > Pulse Counter Module`, independent from
the Modbus module. Opening or using Pulse Counter does not connect, disconnect,
or reconfigure Modbus serial ports.

## Default Configuration
The default lab configuration is:

- Channel: `0`
- Edge: rising
- Pulse value: `0.05`
- Unit: `g`
- Switch frequency: `100 Hz`
- Window origin: `0.0 s`
- Source role: measured device output

These values are configuration inputs, not hard-coded assumptions. Callers can
change the channel, edge selection, pulse value, unit, switch frequency, window
origin, and boundary tolerance.

In the desktop UI, Pulse configuration is saved per stable `Device ID`. Loading
another Device ID loads that device's saved Pulse configuration instead of a
global default.

## DSView CSV Input
The parser expects DSView compressed CSV exports in this shape:

```text
; CSV, generated by libsigrok4DSL ...
; Channels (3/16)
; Sample rate: 25 MHz
; Sample count: ...
Time(s), 0, 1, 2
0,0,0,0
3.74999388,1,0,0
3.75000556,0,0,0
```

Rows are interpreted as change points: each row stores the signal state after
that timestamp. The module extracts configured rising, falling, or both edge
types by comparing adjacent rows.

## Rate Windows And Frequency Switching
The switch frequency defines the fixed rate-window period:

```text
window_s = 1 / switch_frequency_hz
```

With the default `100 Hz`, the module aggregates pulses into fixed `10 ms`
windows aligned to `window_origin_s`. Each window stores:

- start and end time
- pulse count
- quantity
- rate
- cumulative quantity
- boundary pulse count

Boundary pulse count flags pulses within the configured boundary tolerance of a
window edge. These pulses may sit near the contact point between two different
pulse frequencies and should be visible to the operator when reviewing rate
curves.

The module also produces frequency segments from switch-period windows. Each
segment stores pulse count, quantity, mean period, mean frequency, and mean
rate. This is intended for plotting and diagnostics; production pass/fail
limits remain external configuration.

## Calibration Preview
When pulse quantity represents measured device output:

```text
measured_quantity = pulse_count * pulse_value
error_percent = (measured_quantity - standard_quantity) / standard_quantity * 100
new_k = current_k / measured_quantity * standard_quantity
```

These helpers only calculate previews. Any future transmitter K-factor write
must still use the existing Modbus write guard and audit path.

The first UI trial flow stores one Pulse trial with:

- Device ID
- flow point
- trial index
- pulse count
- measured quantity from `pulse_count * pulse_value`
- operator-entered standard quantity
- percent error
- mean rate and boundary pulse count when available

Pulse records use separate Pulse storage tables while remaining linked to the
same stable Device ID used by Modbus.

After saving trials, `Calculate Repeatability...` opens a Pulse-specific
selection dialog. The operator selects one flow point and one consecutive
three-trial window. The module then calculates:

```text
selected_mean_error = (e1 + e2 + e3) / 3
repeatability_stddev = sample standard deviation of the three selected errors
```

The selected-window result is saved as a `pulse_repeatability` operation record
with selected trial IDs, selected trial indexes, flow point, mean error,
repeatability standard deviation, and source trial time range. The module does
not calculate or save repeatability merely because three trials exist.

## Device History
The module-local Pulse history table shows only Pulse records. The application
also provides `History > Device History`, which aggregates records for one
Device ID across modules. It currently combines Modbus operation attempts and
Pulse operation attempts and includes a module filter.

## Future Integration
The intended integration path is:

```text
DSView CSV or live pulse source
  -> pulse_counter edge model
  -> pulse windows and frequency segments
  -> Modbus K-factor or repeatability workflow as an external flow source
  -> existing run artifacts, test records, reports, and guarded write path
```

Do not model pulse input as a Modbus register map. It is a separate external
quantity source that can later be selected by calibration workflows.
